## 1. 产品画像

### 1.1 基本信息

- **产品全称**: 组态王 KingView / 云版组态王 KingView 8.0
- **开发商**: 北京亚控科技发展有限公司（WellinTech, Inc.）
- **总部**: 中国北京
- **成立时间**: 1995 年
- **公司性质**: 民营企业

- **Slogan**: "组态，让工业软件更简单" / "以客为尊，匠心为王"

### 1.2 产品定位与核心价值主张

组态王是中国工业自动化组态软件行业的**开创者和领导者**。其核心价值主张包括：

- **中国第一个自主品牌的工业组态软件**：自 1995 年发布以来，打破国外 SCADA 软件在中国市场的垄断
- **最广泛的设备驱动支持**：号称 6000+ 种设备和协议驱动，覆盖近千个厂家的设备——这是组态王最核心的竞争力
- **中国式定价策略**：价格远低于国际竞争对手（Wonderware、WinCC、iFIX），大幅降低了中国中小型企业的 SCADA 采购门槛
- **全栈产品家族**：从 HMI/SCADA（组态王）到高端 SCADA 平台（KingSCADA）、工业数据库（KingHistorian）、Web 可视化（KingPortal）、工业互联网平台（KingFusion）——形成完整的产品矩阵
- **信创（国产化）适配**：KingView 8.0 云版组态王全面支持国产 CPU（飞腾、鲲鹏等）和国产操作系统（银河麒麟、统信 UOS），响应中国工业软件自主可控战略
- **云时代转型**：从传统的 Windows 组态软件向 WellinOS 云操作系统生态转型，引入功能单元（FU）、云商店、在线更新等概念

- **开发系统（TouchExplorer）**：Windows 桌面应用程序，基于 MFC/C++（推测），提供工程浏览器、画面设计器、脚本编辑器
- **运行系统（TouchView）**：Windows 桌面运行时，负责数据采集、报警处理、画面渲染
- **驱动架构**：每个设备协议编译为独立的 Windows DLL，通过统一的 IO 接口与实时数据库通信

云版组态王的核心架构创新：

- **功能单元（FU, Functional Unit）**：云版组态王的最小可部署单元。每个 FU 是独立的软件模块（采集驱动、报警算法、精灵图、控件、数据处理逻辑等），可独立开发、独立部署、独立更新
- **模型库**：替代传统工程文件。组态配置以数据模型和 FU 模型的形式存储在 WellinOS 模型库中，修改配置无需重启
- **在线组态**：传统组态王的工程文件变化需要重新加载（导致监控中断），云版组态王的配置更改无需 FU 重启
- **集群多副本**：区别于传统双机冗余，每个节点上运行组态王实例，数据区主副副本之间自动同步，故障切换用户无感
- **开放自动化**：内置软 PLC/虚拟 PLC，遵循 IEC 61131-3 标准（IL、ST、SFC、LD、FBD 五种语言）
- **信创适配**：支持飞腾、鲲鹏等国产 CPU，银河麒麟 V10 SP1/SP2/SP3、统信 UOS V2.0 等国产操作系统

### 2.2 HMI 渲染机制

#### 传统版（7.5 SP6）

- **渲染引擎**：Windows GDI+/Direct2D（推测），基于 MFC/Win32 窗口系统
- **画面设计方式**：拖拽式编辑器（Windows 原生 UI），支持点位图、精灵图、基本图素（直线、扇形、椭圆、圆角矩形、折线、多边形）
- **动画系统**：属性动画（颜色变化、位置移动、尺寸变化、可见性切换）
- **图库**：自来水行业 60+ 种 3D 设备图素（每种 15+ 多视角），33 类 2D 图素（每类 10+ 个），总计 3,000+ 图元
- **控件库**：曲线控件、网页控件、立体棒图控件、PID 控件、多媒体动画控件、视频监控控件、ActiveX 控件、条码/二维码控件
- **渲染优化**：7.5 SP6 引入新渲染技术，消除画面毛刺和毛边，实现画面"去粗糙化"
- **分辨率转换**：支持画面分辨率自适应

#### 云版（8.0）

- 基于 KingStudio（界面功能单元编辑器）
- 支持 2D 编辑器，拖拽 2D 设备图素
- 精灵图运行 FU 负责最终效果渲染
- 自定义报表编辑器、UI 组件、图表组件

- 支持通信模式：轮询（Polling）、事件触发、透传
- 驱动持续更新：官网驱动下载区持续发布新驱动（如查阅 2025 年 5-6 月有多达 60+ 个驱动更新）

### 2.4 脚本与表达式

组态王使用**类 C 脚本语言**（非标准 C，为组态王定制）：

- **事件触发**：变量变化事件、画面打开/关闭事件、定时事件、按键事件
- **脚本类型**：
  - 画面脚本（与画面关联）
  - 应用程序脚本（全局）
  - 条件脚本（条件触发）
  - 数据改变脚本（变量值变化时触发）
- **内置函数**：数学运算、字符串处理、文件操作、数据库读写、报表操作、设备通讯等
- **表达式引擎**：支持算术运算、逻辑运算、字符串运算、条件表达式
- **限制**：非通用编程语言（如 Python/JavaScript），学习曲线存在

云版组态王中：
- 报警算法 FU 可使用机器学习算法（结合历史与实时数据）
- 软 PLC 编程遵循 IEC 61131-3 标准（ST/LD/FBD/SFC/IL）

### 2.5 安全模型

传统组态王（7.5 SP6）的安全模型相对基础：
- **用户管理**：本地用户名/密码
- **权限控制**：基于用户级别的功能权限（画面访问、变量读写、参数修改）
- **电子签名**：支持关键操作的电子签名确认
- **二次授权**：部分操作需要二次授权
- **Web 发布安全**：OPC UA 支持数字证书加密传输

云版组态王（8.0）安全增强：
- **集群安全**：多副本架构天然提供高可用和安全隔离
- **信创安全**：基于国产操作系统和 CPU，满足自主可控安全要求
- **云许可**：通过 WellinOS 云端进行许可管理和身份认证

## 3. 功能概览

### 3.1 可视化

- **拖拽式画面编辑**：可视化图形编辑器，拖拽图素到画面
- **图库系统**：内置丰富的行业图库（水处理 3D 图素、通用工业设备、管道阀门、电气符号）
- **精灵图（Sprite）**：可定义动画状态和交互的复合图元
- **控件库**：曲线控件、报表控件、报警控件、多媒体控件、Web 控件、视频监控控件
- **动画效果**：颜色变化、闪烁、移动、缩放、旋转、可见性切换
- **多语言切换**：支持运行时语言切换
- **模板功能**：画面模板可复用

### 3.2 报警管理

- **报警定义**：基于变量阈值的限值报警（高/低/高高/低低/偏差/变化率）
- **报警级别**：多级报警优先级
- **报警记录**：实时报警窗口 + 历史报警查询
- **报警确认**：支持确认注释
- **报警通知**：声音报警、弹窗显示
- **报警分组**：按区域/设备组织
- **报警算法 FU**（云版）：支持行业特有报警算法（如安全生产报警、设备故障预测），可结合传感器检测与阈值、历史数据与机器学习
- **报警输出**：报警信息可写入数据库、发送 MQTT 消息


报警扩展特性：
- **报警抑制**：在设备启停或维护期间，可临时屏蔽特定报警
- **报警联动**：报警触发时可自动执行脚本（如发送短信、切换画面、启动录像）
- **报警统计**：可按时间段、区域、设备统计报警次数和持续时间
- **报警分级延时**：可配置报警触发延时，避免瞬时波动导致的误报
- **报警死区**：配置报警恢复的死区值，防止报警在阈值附近反复触发/消除

云版组态王 8.0 中的报警增强：
- 报警算法以 FU 形式独立运行和更新

## 6. 产品特色

### 6.1 驱动库——万里长城

组态王最核心的差异化优势是其**6000+ 设备驱动库**。这不仅是数量上的优势，更代表了 20+ 年的行业积累：

- **广度**：覆盖 PLC、RTU、智能仪表、变频器、电力规约、楼宇协议、RFID、打印机、扫描枪、称重仪表、气象站等几乎所有工业现场设备类别
- **国产化覆盖**：全面覆盖中国市场上常见的国产 PLC 品牌（信捷、台达、汇川、和利时等），这是国外 SCADA 产品难以匹敌的
- **持续维护**：驱动库持续更新（官网驱动区每月有数十个更新），2025 年 5-6 月即有 60+ 个驱动发布或更新
- **自研 + 开放**：核心驱动自研，同时提供驱动开发工具让设备厂商自行开发

**驱动技术架构深度**：
- 每个驱动编译为独立 Windows DLL，通过统一 IO 接口与实时数据库通信
- 驱动按链路类型分类：串口（COM）、TCP、UDP、USB、PCI、DLL 直连等
- 单个驱动可同时支持多种链路类型（如 ModbusMaster 驱动同时支持 COM 和 TCP）
- 驱动版本独立管理，不受组态王主程序版本限制

**与国外 SCADA 驱动对比**：
- Wonderware：约 500 个驱动，需通过 DA Server 或第三方 OPC Server 扩展
- WinCC：约 500 个驱动，但西门子生态内设备即插即用
- 组态王 6000+ 驱动不依赖 OPC 中间层，直接通信效率更高，但仅限 Windows 平台

## 7. 对 AUDESYS 的参考价值

### 7.1 驱动架构设计

组态王 6000+ 驱动的架构是 AUDESYS HAL 设备驱动设计的最直接参考：

| 组态王驱动设计 | AUDESYS HAL 参考点 |
|-------------|-----------------|
| 每个驱动为独立 DLL，通过统一 IO 接口通信 | AUDESYS HAL Driver 作为独立插件（动态加载），实现统一的 Driver trait |
| 驱动持续独立更新 | AUDESYS 驱动插件独立版本管理和热更新 |
| 设备厂商可自行开发驱动 | AUDESYS 提供 Driver SDK，开放第三方驱动开发 |
| 驱动按协议类型分类（串口/TCP/UDP/USB） | AUDESYS 驱动接口抽象链路层（Serial、TCP、UDP、CAN 等） |

**AUDESYS HAL 驱动接口设计建议**：

```
// 参考组态王驱动分层思想
trait HalDriver {
    fn connect(&mut self, config: DriverConfig) -> Result<(), Error>;
    fn disconnect(&mut self) -> Result<(), Error>;
    fn read_signal(&self, signal_id: SignalId) -> Result<SignalValue, Error>;
    fn write_signal(&mut self, signal_id: SignalId, value: SignalValue) -> Result<(), Error>;
    fn subscribe(&self, signal_id: SignalId, callback: Box<dyn Fn(SignalValue)>) -> Result<(), Error>;
}

// 链路层抽象（参考组态王的多种链路类型）
enum LinkType {
    Serial { port: String, baud: u32, data_bits: u8, parity: Parity, stop_bits: u8 },
    Tcp { host: String, port: u16 },
    Udp { host: String, port: u16 },
    Can { interface: String },
    Pci,
    Usb { vendor_id: u16, product_id: u16 },
}
```

### 7.2 在线组态——AUDESYS 运行时热更新

组态王云版的在线组态（配置变更无需重启）是 AUDESYS Runtime 的关键参考：

- **传统模型**：配置变更 → 重启 Runtime → 服务中断 → 生产影响
- **目标模型**：配置变更 → Config Barrier（配置屏障）→ RT 周期边界应用 → 无中断

这与 AUDESYS 已有的 D17 决策（Config Barrier + LockLevel）完全吻合。组态王云版通过 FU 模型+数据模型分离实现了类似的能力，AUDESYS 可以在 HAL 配置管理层面借鉴。

### 7.3 多协议数据采集的中间件抽象

组态王需要同时与 Modbus、S7、OPC UA、MQTT、电力规约等多种协议通信，其"统一 IO 接口 + 独立驱动 DLL"的架构模式与 AUDESYS HAL 的 amw（AUDESYS Middleware）抽象层设计理念一致：

- 组态王的统一 IO 接口 ↔ AUDESYS 的 amw Transport trait
- 组态王的设备驱动 DLL ↔ AUDESYS 的 amw Implementation（amw_inproc、amw_zenoh 等）
- 组态王的对外数据接口（OPC UA Server、MQTT、WebService）↔ AUDESYS 的 OPC UA Server + MQTT Bridge 规划

### 7.5 B/S vs C/S 架构抉择

组态王的 C/S（传统版）→ B/S（云版）演进路径为 AUDESYS Studio 提供了明确的参考：

- **传统版 C/S**（TouchExplorer + TouchView）：成熟稳定，但依赖 Windows、客户端需安装
- **云版 B/S**（KingStudio 浏览器化 + FU 运行时）：零安装、跨平台，但功能可能不如桌面版完善

**AUDESYS 的推荐策略**：
- **Phase 1**：桌面优先（Electron 封装），保证核心功能（PLC 编程、HMI 设计、仿真调试）
- **Phase 2**：关键模块 Web 化（HMI 设计器、监控画面运行时）
- **Phase 3**：全栈 Web 化（含 PLC 编程的 Web IDE）

### 7.6 从组态到开放自动化

组态王 8.0 引入软 PLC/虚拟 PLC 的路径与 AUDESYS 的 Runtime 定位高度吻合：

| 组态王 8.0 能力 | AUDESYS 对应能力 |
|----------------|----------------|
| 虚拟 PLC（IEC 61131-3） | AUDESYS Runtime（控制程序执行引擎） |
| FU 从云端部署到现场 | AUDESYS Runtime 远程部署和管理 |
| FU 独立运行和更新 | AUDESYS 模块热加载 |
| 云商店交易 FU | AUDESYS Marketplace（插件/模板/驱动） |

### 7.7 生态策略——"让 4000 万自动化工程师成为开发者"

组态王云商店的"社会化分工"理念为 AUDESYS 的生态建设提供了启发：

- 组态王吸引设备厂商自研驱动 → AUDESYS 吸引硬件厂商适配 HAL 驱动
- 组态王云商店允许交易 FU → AUDESYS Marketplace 允许交易插件、模板、HMI 组件
- 组态王降低开发门槛（KingStudio） → AUDESYS Studio 降低工业软件开发门槛

### 7.8 信创/国产化路径

如果 AUDESYS 未来考虑中国市场，组态王在信创适配方面的经验具有参考价值：
- 操作系统适配：Linux（银河麒麟、统信 UOS）
- CPU 适配：ARM64（飞腾、鲲鹏）、x86_64
- 审计支持：国产加密算法（SM2/SM3/SM4）

### 7.9 需警惕的陷阱

1. **Windows 依赖**：传统组态王深度绑定 Windows（MFC/GDI+/DLL 架构），Web 化转型困难。AUDESYS 应从第一天就跨平台设计。
2. **闭源驱动的维护负担**：6000+ 驱动由一家公司维护，随着设备更新换代，维护成本极高。AUDESYS 应优先支持标准协议（OPC UA、Modbus、MQTT），鼓励社区贡献驱动。
3. **类 C 脚本语言的限制**：非标准脚本语言导致的学习成本和可移植性问题。AUDESYS 应选择通用语言（Python/Lua/JavaScript）或支持多语言。
4. **云版与旧版的断层**：KingView 7.5 SP6 和 8.0 的架构完全不同，老用户迁移成本高。AUDESYS 应在架构设计早期就考虑向后兼容策略。

> AUDESYS 不需要重复组态王的 Windows 时代，而应直接跳到云原生、跨平台、多语言、信创兼容的新一代架构。
> 这正是后发优势的体现——站在 30 年行业积累的肩膀上，用现代技术栈重新定义工业 SCADA 平台。
> 这需要勇气，但不需要从头摸索——因为组态王已经用三十年时间验证了哪些功能是工业现场真正需要的。
