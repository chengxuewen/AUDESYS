# Forklift PLC

## 1. 产品画像

### 1.1 产品全称与开发商

- **产品全称**: Forklift PLC（轻量级嵌入式软 PLC 运行时）
- **开发商/组织**: 由 **Forklift 社区** 维护，最初由嵌入式系统开发者社区发起（https://github.com/forklift-plc）

### 1.2 产品定位与核心价值主张

Forklift PLC 定位为 **面向资源受限 MCU 的轻量级开源软 PLC 运行时**，其核心价值主张是：

1. **极小的内存占用** — 设计目标是在数十 KB RAM 和数百 KB Flash 的 MCU 上运行完整的 PLC 运行时
2. **确定性执行** — 通过最小化 RTOS 占用实现可预测的扫描周期
3. **部分 IEC 61131-3 支持** — 支持常用的 IEC 61131-3 语言子集（ST 和 LD）
4. **嵌入式原生** — 专为 ARM Cortex-M 系列 MCU 设计，原生支持 STM32、GD32 等
5. **开源透明** — 完整的运行时源代码，可审计和自定义

## 2. 技术特性

### 2.1 核心架构

Forklift PLC 采用 **极简分层架构**，专为资源受限的 MCU 环境优化：

```
+--------------------------------------------------+
|              Forklift PLC Runtime                |
+--------------------------------------------------+
|  +--------------------------------------------+  |
|  |  Application Layer (IEC 61131-3 程序)       |  |
|  |  - ST 程序逻辑                               |  |
|  |  - LD 梯形图逻辑（预编译）                     |  |
|  +--------------------------------------------+  |
|  +--------------------------------------------+  |
|  |  Runtime Engine                             |  |
|  |  - PLC State Machine                        |  |
|  |  - Scan Cycle Manager                       |  |
|  |  - Task Scheduler (Cyclic Only)             |  |
|  |  - I/O Image Table Manager                  |  |
|  +--------------------------------------------+  |
|  +--------------------------------------------+  |
|  |  HAL Abstraction Layer                      |  |
|  |  - GPIO Driver                              |  |
|  |  - ADC Driver                               |  |
|  |  - PWM Driver                               |  |
|  |  - UART/Modbus Driver                       |  |
|  +--------------------------------------------+  |
|  +--------------------------------------------+  |
|  |  RTOS / Bare-Metal Layer                    |  |
|  |  - FreeRTOS / RT-Thread / Bare-Metal        |  |
|  |  - Timer Interrupt                         |  |
|  |  - Memory Management                       |  |
|  +--------------------------------------------+  |
+--------------------------------------------------+
```

### 2.2 关键技术能力

#### 极简扫描周期

Forklift PLC 的扫描周期模型经过极简化，适合 MCU 环境：

```
1. Read Inputs:    从 GPIO/ADC 寄存器直接读取输入
2. Execute Logic:  执行预编译的 PLC 程序
3. Write Outputs:  将结果写入 GPIO/PWM 寄存器
4. Timer Wait:     使用硬件定时器等待下一周期
```

- **典型周期时间**: 1-10ms（STM32F4 系列）
- **调度方式**: 硬件定时器中断驱动
- **最小周期**: 可低至 500us（取决于程序复杂度）

#### 资源占用

| 资源 | 典型占用 | 说明 |
|------|---------|------|
| Flash | 20-60 KB | 运行时 + 标准库 |
| RAM | 4-16 KB | I/O 映像表 + 栈空间 |
| 堆栈 | 1-4 KB | 任务栈 + 中断栈 |
| CPU 占用 | 10-30% | 典型扫描周期负载 |

#### 类型系统

Forklift PLC 支持 IEC 61131-3 类型子集（资源受限优化）：

| 类型 | 位宽 | 说明 |
|------|------|------|
| BOOL | 1 bit | 布尔类型 |
| BYTE | 8 bit | 字节 |
| INT | 16 bit | 有符号整型 |
| UINT | 16 bit | 无符号整型 |
| DINT | 32 bit | 双字有符号整型 |
| REAL | 32 bit | IEEE 754 浮点（可选） |
| TIME | 32 bit | 毫秒精度时间 |

#### 编程语言支持

| 语言 | 缩写 | 支持状态 | 说明 |
|------|------|---------|------|
| Structured Text | ST | 完全支持 | 主要的编程语言 |
| Ladder Diagram | LD | 有限支持 | 通过转换工具预编译 |
| Function Block Diagram | FBD | 不支持 | 资源受限 |
| Instruction List | IL | 不支持 | 已退役 |
| Sequential Function Chart | SFC | 不支持 | 资源受限 |

### 2.3 支持的硬件平台

Forklift PLC 主要面向 ARM Cortex-M 系列 MCU：

| 平台 | 架构 | 典型配置 | 扫描周期 |
|------|------|---------|---------|
| STM32F103 | Cortex-M3 | 72MHz, 20KB RAM, 128KB Flash | 5-10ms |
| STM32F4/F7 | Cortex-M4F | 168MHz, 128KB RAM, 512KB Flash | 1-5ms |
| STM32L4 | Cortex-M4 | 低功耗系列 | 5-20ms |
| GD32 系列 | Cortex-M3/M4 | 国产替代 | 5-10ms |
| ESP32 | Xtensa LX6 | 240MHz, 520KB RAM | 2-10ms |
| Arduino Due | Cortex-M3 | 84MHz | 10-20ms |

### 2.4 开发与部署流程

```
PC 端开发环境
  |
  | 1. 使用 ST 编写 PLC 程序
  | 2. 交叉编译为 MCU 目标代码
  | 3. 通过串口/JTAG 下载
  |
  v
MCU 端 Forklift PLC Runtime
  |
  | 4. 启动运行时初始化
  | 5. 加载 PLC 程序到内存
  | 6. 进入 RUNNING 状态
  | 7. 开始扫描周期循环
```

## 6. 产品特色

### 6.1 独特优势

1. **极低资源占用**: 能够在仅有 20KB RAM 和 128KB Flash 的 MCU 上运行完整的 PLC 运行时
2. **确定性执行**: 无操作系统或最小 RTOS 依赖，硬件定时器直接驱动扫描周期
3. **嵌入式原生设计**: 从零开始为 ARM Cortex-M 设计，而非从 Linux 移植
4. **低成本方案**: 硬件成本可低至 2-5 美元（STM32 国产替代芯片）

### 6.2 设计理念

- **"最小可行 PLC"**: 移除所有非必要的功能，只保留核心 PLC 运行时
- **"MCU 为第一目标"**：所有设计决策都从资源受限 MCU 的角度出发
- **"确定性优先"**：可预测的执行时间优先于功能丰富性

## 7. 对 AUDESYS 的参考价值

### 7.1 可借鉴的架构设计

#### 1. 资源受限下的运行时设计

Forklift PLC 展示了在 MCU 资源限制下设计 PLC 运行时的关键约束：

| 资源约束 | Forklift PLC 方案 | AUDESYS 参考 |
|---------|------------------|-------------|
| Flash 空间 | 20-60 KB 运行时 | 嵌入式 HAL 的代码量控制 |
| RAM 限制 | 4-16 KB 映像表 | 最小化 I/O 缓冲区 |
| 无 MMU | 平坦内存模型 | 嵌入式 HAL 的内存管理 |
| 无文件系统 | 编译时链接 | 嵌入式配置的编译时处理 |

#### 2. 最小化 RTOS 占用

Forklift PLC 在 RTOS 使用上采用极简策略：

- 仅使用 RTOS 的定时器和任务调度功能
- 避免使用动态内存分配、消息队列、信号量等重量级 RTOS 组件
- 扫描周期通过硬件定时器中断直接驱动，不经过 RTOS 调度器

**AUDESYS 参考**: 对于嵌入式 HAL 和轻量级 Runtime，可参考 Forklift PLC 的 RTOS 最小化策略，避免不必要的 RTOS 抽象层。

#### 3. 硬件抽象层（HAL）设计

Forklift PLC 的 HAL 层设计极为简洁，仅包含最必要的 I/O 抽象：

```
HAL Interface:
  - digitalRead(pin) -> BOOL
  - digitalWrite(pin, value)
  - analogRead(channel) -> INT
  - analogWrite(channel, value)
  - serialRead(port) -> BYTE
  - serialWrite(port, data)
```

**AUDESYS 参考**: Forklift PLC 的 HAL 简洁性提醒 AUDESYS HAL 设计避免过度工程化。虽然 AUDESYS HAL 需要完整的通信原语（Signal/StreamChannel/RPC），但嵌入式 HAL 子集可以借鉴 Forklift PLC 的极简接口设计。

| 维度 | Forklift PLC | AUDESYS |

| 核心定位 | MCU 嵌入式软 PLC | 工业控制系统模拟平台 |

**互补关系**:
- Forklift PLC 展示了 **MCU 级别 PLC 运行时的最小实现**，对 AUDESYS 嵌入式 HAL 和轻量级 Runtime 设计有直接参考价值
- AUDESYS 的 HAL 设计（Signal/StreamChannel/RPC）提供 Forklift PLC 所缺少的通信抽象能力
- Forklift PLC 的 **确定性执行策略**（硬件定时器驱动）对 AUDESYS RT 调度设计有参考意义
- Forklift PLC 的 **资源受限经验** 可为 AUDESYS 的嵌入式部署场景提供优化指导

### 7.3 对 AUDESYS 嵌入式 HAL 的启示

Forklift PLC 的嵌入式设计经验可直接应用于 AUDESYS 的嵌入式 HAL 设计：

1. **静态内存分配**: 所有数据结构在编译时确定大小，避免运行时动态分配
2. **中断驱动 I/O**: 使用 MCU 外设中断直接驱动 I/O，减少轮询开销
3. **编译时配置**: 尽可能在编译时确定配置参数，减少运行时配置开销
4. **最小依赖**: 零外部依赖或仅依赖 CMSIS/Cube HAL 等 MCU 标准库
5. **可预测性能**: 所有代码路径的执行时间可预先计算

### 2.5 扫描周期详解

Forklift PLC 的扫描周期管理是资源受限环境下的关键设计。与 Linux 上的 OpenPLC 使用 clock_nanosleep 不同，MCU 环境使用硬件定时器中断驱动：

```
硬件定时器中断
    |
    v
Timer ISR -> 设置扫描周期标志 -> 退出中断
    |
    v
主循环检测到标志
    |  1. Read Inputs:    从 GPIO 寄存器组批量读取数字输入
    |                     从 ADC 寄存器读取模拟输入通道
    |  2. Execute Logic:  执行预编译的 PLC 用户程序（函数调用）
    |  3. Write Outputs:  将结果写入 GPIO 输出寄存器
    |                     更新 PWM 占空比寄存器
    |  4. Wait:           等待下一个定时器中断
    v
下一个周期
```

**关键时序参数**:

| 参数 | 典型值 | 说明 |
|------|--------|------|
| 周期时间 | 1-10ms | 可配置，由硬件定时器决定 |
| 抖动 (Jitter) | +/- 5-50us | 取决于中断优先级和代码执行时间 |
| 输入采样延迟 | < 1us | 直接从寄存器读取 |
| 输出更新延迟 | < 1us | 直接写入寄存器 |

### 2.6 内存管理策略

Forklift PLC 的内存管理是针对 MCU 环境专门优化的：

**静态内存分配**:
- 所有 I/O 映像表在编译时分配
- 用户程序变量在链接时分配固定地址
- 无 malloc/free 动态内存分配

**内存布局**:
```
+------------------+  <-- RAM Start
| I/O Image Table  |  数字输入/输出映像表 (2-4 KB)
|   - %IX 区域     |  数字输入
|   - %QX 区域     |  数字输出
|   - %IW 区域     |  模拟输入
|   - %QW 区域     |  模拟输出
+------------------+
| User Variables   |  PLC 程序变量区 (4-8 KB)
|   - 全局变量     |
|   - 局部变量     |
|   - 保持变量     |
+------------------+
| Runtime Heap     |  运行时内部数据结构 (1-4 KB)
+------------------+
| Stack            |  任务栈 + 中断栈 (1-2 KB)
+------------------+  <-- RAM End

+------------------+  <-- Flash Start
| Bootloader       |  启动代码 (4-8 KB)
+------------------+
| Runtime Core     |  PLC 运行时核心 (20-40 KB)
+------------------+
| User Program     |  预编译的 PLC 用户程序 (4-16 KB)
+------------------+
| Standard Library |  IEC 标准库实现 (8-16 KB)
+------------------+  <-- Flash End
```

### 2.7 编译与代码生成

Forklift PLC 的编译流程不同于 Beremiz 和 OpenPLC 的运行时编译模式：

```
PC 端交叉编译流程:
  ST 源文件 (.st)
    |
    | [Forklift ST 编译器 - 桌面端工具]
    |  词法分析 -> 语法分析 -> AST -> 优化 -> C 代码生成
    |
    +-> C 源代码 (.c)
         |
         | [ARM GCC 交叉编译器]
         |  arm-none-eabi-gcc -mcpu=cortex-m4 -Os
         |
         +-> 目标二进制 (.bin / .elf / .hex)
              |
              | [通过串口/JTAG/SWD 下载]
              |
              v
         MCU Flash 执行
```

**关键特性**:
- 编译在 PC 端完成，MCU 只执行预编译的二进制代码
- 编译器针对资源受限环境优化：-Os 优化，避免函数调用开销
- 生成的代码使用静态函数调用，不使用函数指针或动态加载
- 运行时初始化时直接调用生成的初始化函数和循环函数

### 2.8 中断处理模型

Forklift PLC 的中断处理模型是 MCU 实时性的关键：

```
中断优先级分层:
  级别 0 (最高): 硬件定时器 - 扫描周期同步
  级别 1:        外部中断 - 紧急输入信号
  级别 2:        UART 接收 - Modbus 通信
  级别 3:        ADC 转换完成 - 模拟采样
  级别 4 (最低): 普通 I/O 中断

扫描周期与中断的关系:
  - 定时器中断 ISR 只设置标志，不执行 PLC 逻辑
  - 主循环在中断上下文中不可阻塞
  - 外部中断输入被缓存到映像表，由主循环处理
  - Modbus 通信使用 DMA + 环形缓冲区，不阻塞主循环
```

## 3. 功能概览（续）

### 3.4 PLC 状态机详细

Forklift PLC 的状态机相比标准 PLC 做了简化：

```
POWER_ON
    |
    v
  INIT ------------> ERROR (初始化失败)
    |
    v
  STOPPED <--------> RUNNING (通过控制命令切换)
    |
    v
  ERROR -----------> STOPPED (错误恢复)
```

| 状态 | 描述 | 进入条件 |
|------|------|---------|
| INIT | 上电初始化，硬件和运行时初始化 | 系统上电 |
| STOPPED | 停止状态，等待启动命令 | 初始化完成或错误恢复 |
| RUNNING | 正常运行，执行扫描周期 | 收到启动命令 |
| ERROR | 错误状态，停止程序执行 | 运行时错误或看门狗超时 |

### 3.5 看门狗机制

Forklift PLC 使用硬件看门狗（IWDG）确保系统可靠性：

- **硬件看门狗**: MCU 内置独立看门狗定时器
- **超时时间**: 2-3 倍扫描周期时间
- **喂狗位置**: 扫描周期完成时喂狗
- **超时动作**: 系统复位
- **调试模式**: 支持在看门狗超时前暂停

### 3.6 通信协议实现

#### Modbus RTU 协议栈

Forklift PLC 的 Modbus RTU 实现是专为 MCU 优化的轻量级实现：

## 6. 产品特色（续）

### 6.3 嵌入式 PLC 设计哲学

Forklift PLC 代表了一类嵌入式 PLC 的设计哲学，与桌面 PLC 有本质区别：

**"静态优于动态"**:
- 编译时决定所有变量地址和内存布局
- 运行时无动态加载、无插件系统、无反射
- 确定性优于灵活性

**"硬件定时器优于软件调度"**:
- 使用 MCU 硬件定时器直接驱动扫描周期
- 不依赖操作系统调度器
- 中断延迟直接影响周期精度

**"裸机优于 RTOS"**:
- 最小化 RTOS 功能使用
- 尽可能使用裸机（bare-metal）运行
- 减少上下文切换的开销和不确定性

**"预先编译优于运行时编译"**:
- PLC 程序在 PC 端交叉编译
- MCU 只执行预编译的二进制代码
- 无运行时解析、编译、动态加载开销

### 6.4 与大型 PLC 运行时的关键差异

| 维度 | 大型 PLC 运行时 | Forklift PLC |
|------|---------------|-------------|
| 操作系统 | Linux / Windows | 裸机 / FreeRTOS |
| 内存 | MB 级别 | KB 级别 |
| 程序加载 | 动态加载 .so/.dll | 编译时链接 |
| 通信栈 | 完整的协议栈 | 精简 Modbus |
| 调试 | 全面的调试接口 | 基本的串口调试 |
| 热更新 | 支持 | 不支持 |
| 安全 | TLS/JWT | 无 |
| 文件系统 | 支持 | 无 |

## 7. 对 AUDESYS 的参考价值（续）

### 7.4 嵌入式 Runtime 设计原则

从 Forklift PLC 的设计中，可以提炼出嵌入式 PLC 运行时设计的核心原则，这些原则对 AUDESYS 的嵌入式 HAL 设计有直接参考价值：

**原则 1: 编译时确定一切**
- 变量地址、内存布局、I/O 映射都在编译时确定
- 运行时零配置、零解析
- 类比：AUDESYS 的 Config Barrier + LockLevel 设计

**原则 2: 中断是唯一的实时保证**
- 硬件定时器中断决定扫描周期精度
- 中断 ISR 尽可能短，仅在主循环中执行 PLC 逻辑
- 类比：AUDESYS 的 RT 线程设计（SCHED_FIFO）

**原则 3: 最小化协议栈**
- Modbus RTU 是嵌入式 PLC 的基本通信协议
- 不需要完整的 TCP/IP 栈（除非必需）
- 类比：AUDESYS Phase 1 选择 Modbus RTU/TCP + HART

**原则 4: 可预测性优先于功能丰富性**
- 每个代码路径的执行时间可预先计算
- 避免动态内存分配和递归调用
- 类比：AUDESYS RT 路径的确定性要求

### 7.5 Forklift PLC 与 AUDESYS 嵌入式 HAL 的映射

| Forklift PLC 概念 | AUDESYS 嵌入式 HAL 对应概念 |
|------------------|---------------------------|
| GPIO 驱动 | HalTransport 的 Signal 原语 |

| Modbus RTU 协议 | D23 的 Modbus 协议适配 |
| 静态 I/O 映像表 | HAL 的 %IX/%QX/%IW/%QW 地址映射 |
| 看门狗机制 | PLC 运行时的 Watchdog 策略 |
| 串口调试 | 工业调试桥（规划中） |
| 裸机/FreeRTOS | amw_inproc 传输层（Phase 1） |

### 7.7 嵌入式 PLC 运行时的未来方向

Forklift PLC 代表的嵌入式软 PLC 方向与 AUDESYS 的轻量级 Runtime 设计有共同的未来趋势：

### 6.8 安全考虑

嵌入式 PLC 的安全考虑与传统 IT 安全有显著区别：

- **物理安全优先**: 嵌入式 PLC 运行在受控的物理环境中，网络攻击面有限
- **无远程管理**: 大多数部署场景无远程管理需求
- **看门狗是主要安全机制**: 硬件看门狗保证系统在故障时自动恢复
- **固件加密**: 可以通过 MCU 的读保护（RDP）防止固件逆向
- **Modbus 无认证**: Modbus RTU 协议本身无认证机制，依赖物理隔离
