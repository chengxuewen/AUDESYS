# 和利时 Hollysys（HOLLiAS MACS-K / IC）

## 1. 产品画像

### 1.1 公司基本信息

- **全称**: 和利时集团（Hollysys Group）/ 北京和利时系统工程有限公司

- **成立时间**: 1993 年（以电力行业自动化控制科研团队起家）

- **DCS 安装案例**: 超过 20,000 例

### 1.2 产品定位与核心价值主张

和利时是中国工业自动化领域"三足鼎立"格局（中控/和利时/国电智深）的重要参与者，其核心价值主张是：

- **跨行业平台战略**：在同一技术平台（HOLLiAS MACS）上覆盖电力、石化、铁路、核电四大关键行业，这一策略在国内外 DCS 厂商中极为罕见
- **内生安全理念**：从控制芯片、操作系统到应用层的全栈安全可控设计，国内首个"内生安全 DCS 控制系统"
- **铁路信号技术独特性**：中国高铁列控系统核心供应商（LKJ 列车运行监控、ATP 车载设备、RBC 无线闭塞中心），这是其他 DCS 厂商不具备的能力
- **核电仪控国产化**：通过子公司广利核（CTEC）为"华龙一号"核电机组提供汽轮机控制系统（TCS，Turbine Control System）
- **开放式集成平台**：HOLLiAS Bridge 通用互联平台，支持 Profibus-DP/PA、HART、Modbus、OPC、IEC 101-104 等多种协议

#### MACS-K 网络层级详解

| 网络层 | 名称 | 协议 | 速率 | 拓扑 | 冗余 |
|--------|------|------|------|------|------|
| MNET（管理网） | Management Network | TCP/IP | 100/1000Mbps | 星型/环型 | 可选 |
| SNET（系统网） | System Network | TCP/IP, UDP | 100/1000Mbps | 星型/环型/总线型 | 双冗余 |
| INNER-BUS | 内部总线 | 100M 专用以太网 | 100Mbps | 星型（通过 Opal8 交换机） | 双冗余 |
| CNET/IO-BUS | Control Network | Profibus-DP | 1.5Mbps（默认）/ 500kbps / 187.5kbps | 星型或总线型 | 双冗余 |

**AUDESYS 关联分析**：
- SNET → **StreamChannel**：操作站与控制站之间的实时数据订阅/发布
- IO-BUS → **Signal**：控制器周期性轮询 I/O 模块获取最新采样值
- INNER-BUS → 内部通信（K-CU03 与 K-BUS04 之间），属于 MACS-K 内部实现细节，HAL 层面无需直接对应

### 2.3 冗余模型（与 ECS-700 对比）

MAC-K 的冗余设计与中控 ECS-700 相似但存在细节差异：

| 冗余层级 | MACS-K 实现 | 与 ECS-700 的主要差异 |
|---------|-------------|-------------------|
| 供电冗余 | 系统电源 + 现场电源 + 辅助电源三电源隔离，负载均衡冗余 | ECS-700 为双路 24VDC；MACS-K 额外增加了三电源隔离设计 |
| 控制器冗余 | 主从热备（Master/Slave Hot Standby），切换 < 3 个 IEC 周期 | ECS-700 切换时间 < 1 个扫描周期（更快） |
| 网络冗余 | SNET A/B + INNER-BUS A/B + IO-BUS A/B 全三重网络冗余 | ECS-700 为 SCnet + E-BUS + L-BUS 三层；MACS-K 多了 INNER-BUS 层 |
| I/O 模块冗余 | 成对插入冗余底座，无需额外配置 | 两者类似，均支持热插拔 |
| 热电偶/热电阻冗余 | 支持 TC/RTD 冗余（行业领先） | ECS-700 标准配置不原生支持 TC/RTD 冗余（需要额外模块） |
| 安全交换机 | HollySys Safety Switch（仅允许 DCS 数据包通过） | ECS-700 使用通用工业交换机 + 控制器内置防火墙 |
| 操作安全 | 可配置到画面级、操作指令级、设备级授权 | 与 ECS-700 类似 |

### 2.6 编程语言与工程环境

MACS-K 完全符合 **IEC 61131-3** 标准：

| 语言 | 用途 |
|------|------|
| CFC（Continuous Function Chart） | 连续过程控制（类似 FBD，但更自由布局） |
| SFC（Sequential Function Chart） | 顺序控制、批处理配方 |
| LD（Ladder Diagram） | 逻辑联锁、离散量控制 |
| ST（Structured Text） | 复杂计算、数据处理 |

**工程开发环境（HOLLiAS MACS V6）**：
- **AutoThink**：集成化工程软件套件（项目中心 + 仿真器 + 图形编辑 + AutoThink 编程）
- **多工程师协同组态**：最多 10 人同时在同一个项目上工作
- **硬件配置自动生成**：减少手动组态出错
- **虚拟工厂仿真**：完全等同真实控制逻辑和 HMI 界面的 PC 端仿真
- **在线修改**：控制逻辑在线修改、图形在线修改、硬件在线修改——全部无扰
- **全面诊断**：覆盖控制器冗余、网络状态、I/O 通道短路、断线、超量程、机柜温度、电源报警等

### 2.7 HiaGuard 安全仪表系统

HiaGuard 是国内首套拥有自主知识产权的 SIS：

- **架构**：三重化冗余容错（TMR），支持 3-2-0 降级模式
- **认证**：TÜV Rheinland IEC 61508 SC3 / SIL3（2012 年认证）
- **组态工具**：Safe-AutoThink (Safe-AT)，作为 T3 工具通过 TÜV SIL3 认证
- **编程语言**：受限 FBD / LD（安全编译路径）+ 非受限 ST（复杂算法路径）
- **系统规模**：支持 15 域，每域 64 控制站，单站 1,856 I/O 点
- **与 DCS 集成**：通过安全以太网与 HOLLiAS MACS DCS 共享 HMI
- **系统可用性**：1+1 系统电源/现场电源冗余、I/O 模块冗余

### 2.10 MACS IC 全自主可控架构

MACS IC 是面向"去美化"需求的特殊版本：

- **全国产芯片**：处理器、存储、通信芯片全部国产
- **全国产操作系统**：基于国产操作系统（银河麒麟或统信UOS，待确认）
- **软件跨平台**：一套源代码适配多种操作系统（源代码级跨平台编译）
- **内生安全**：主控单元内置独立可信计算安全防护系统
- **首台套验证**：大唐洛河电厂600MW超超临界机组 DCS+DEH 一体化改造（2022年1月）
- **规模能力**：支持128域，每域1,200,000标签，64控制站，单站360 IO设施（远超标准MACS-K）

## 6. 产品特色

### 6.1 跨行业平台统一——四行业共享同一技术底座

这是和利时最核心的差异化特征。MACS 平台同时支撑：

- HOLLiAS MACS-K（工业 DCS）
- HOLLiAS-N（核电仪控）
- HiaGuard（安全仪表系统）
- TIS-HS 列控联锁一体化（铁路信号 SIL4）

虽然不同行业的具体硬件和软件有所差异，但核心的**安全计算机架构、冗余机制、通信协议栈、工程工具链**在不同行业间高度复用。这种"平台化+行业适配"的模式降低了总研发成本，同时形成了跨行业的竞争力壁垒。

### 6.2 内生安全 DCS——首个"自免疫"控制系统

MACS IC 的全自主可控不仅是芯片/操作系统的国产替代，更强调"内生安全"理念：

- **内置安全可信防护体系**：主控单元内置独立的可信计算安全防护系统
- **安全审计加密**：内置多种安全审计和数据加密算法
- **等保三级合规**：满足中国信息安全等级保护三级要求
- **OT/IT 深度融合**：从控制层到管理层的全链路安全可信
- **全国产化**：芯片、数据库、操作系统、工业软件、自动化技术全部国产

这与中控的"自主可控+AI 赋能"路线不同：和利时更强调安全合规属性，适合对安全等级要求极高的核电、电力、铁路行业。

### 6.3 工业光总线控制（OCS）——物理层创新

2025 年发布的 OCS（Optical Control System）代表了和利时的技术原创性：

- **Onet 工业光总线技术**：用光纤取代铜缆，实现全光通信
- **软件定义 I/O**：I/O 通道功能由软件定义而非硬件固定
- 基于 HOLLiAS MACS 平台，非全新架构

这与中控的 UCS 思路不同：中控是架构级革命（"去控制柜"），和利时是物理层升级（"去铜缆"）。

### 6.4 铁路信号技术沉淀——SIL4 安全认证能力

和利时在铁路信号领域的技术积累是全球 DCS 厂商中独一无二的：

- SIL4 安全完整性等级：要求每小时危险故障概率 < 10⁻⁹，远超过一般工业的 SIL3（< 10⁻⁷）
- 故障-安全原则（Fail-Safe）：任何故障均导向安全侧
- 二乘二取二架构（2oo2×2）：两系独立运算+输出比较+故障诊断
- CTCS-2/3 级兼容：支持既有线提速和高铁新建线路
- ERTMS/ETCS 兼容能力：满足国际铁路信号标准

**对 AUDESYS 的意义**：SIL4 级别的安全关键系统是 HAL 安全设计的上限参考。虽然 AUDESYS Phase 1 不需要 SIL4 认证，但 Security Domain 的隔离设计和故障模式处理可以借鉴铁路信号的 Fail-Safe 哲学。

### 6.5 核电仪控国产化突破

这是和利时近年的战略突破：

- 通过广利核公司（CTEC）与中广核深度绑定
- 华龙一号核电汽轮机控制保护系统（TCS）：我国首个自主核电 TCS
- HOLLiAS-N 平台：与主 DCS 平台（和睦系统 FirmSys）一体化设计
- 从"首台套"迈向"再进化"：在三澳项目持续优化

核电仪控的门槛极高：需要核安全级抗震能力、辐射环境可靠运行、严格的软件 V&V（验证与确认）流程。和利时在这一领域的成功证明了其技术平台的可靠性。

### 6.6 三电源隔离设计

MACS-K 的供电设计是与 ECS-700 的一个重要差异点：

- 系统电源（System Power）：为控制器的数字电路和通信电路供电
- 现场电源（Field Power）：为与仪表相连的 I/O 电路供电
- 辅助电源（Auxiliary Power）：为机柜风扇、照明等辅助设备供电

三电源隔离的好处：
- 现场来的干扰不会影响数字电路和通信
- 辅助设备故障不会影响控制运行
- 维护时可以单独切断辅助电源而不影响控制

## 7. 对 AUDESYS 的参考价值

### 7.1 HAL 通信模型与 MACS-K 内部通信对照

MACS-K 的三层网络（SNET/INNER-BUS/IO-BUS）与 AUDESYS HAL 三原语的对应关系：

| HAL 原语 | MACS-K 对应通信 | 特征 | 与 ECS-700 对照的差异 |
|---------|----------------|------|---------------------|
| **Signal** | IO-BUS（控制器 ↔ I/O 模块轮询） | RS485 物理层，Profibus-DP 协议，1.5Mbps | 与 ECS-700 的 L-BUS 类似，但物理层不同 |
| **StreamChannel** | SNET（操作站订阅控制站实时数据） | TCP/IP 上构 建的实时数据流，100/1000Mbps | 与 ECS-700 的 SCnet 类似 |
| **RPC** | 操作站→控制站下行指令 | 通过 SNET 承载 | 与 ECS-700 相同 |

**关键差异点**：

1. **INNER-BUS 的引入**：MACS-K 在主控制器（K-CU03）与 IO-BUS 模块（K-BUS04）之间引入了 INNER-BUS（100M 专用以太网交换机）。这是 MACS-K 独有的网络层，ECS-700 没有此层。对 AUDESYS 而言，INNER-BUS 是系统内部实现细节，HAL 不需要直接建模此层。

2. **IO-BUS 的 Profibus-DP 协议绑定**：MACS-K 的 IO-BUS 直接绑定 Profibus-DP 协议，而 ECS-700 使用自定义确定性以太网协议。这意味着 AUDESYS 如果要模拟 MACS-K 的 I/O 通信，需要支持 Profibus-DP 主站协议（作为 amw_transport 的一种实现）。

3. **SNET 的安全增强**：HollySys Safety Switch 仅在 SNET 层实现数据包过滤。这提示 AUDESYS 的 StreamChannel 可能需要支持**传输层安全过滤**（在 amw_transport 层面而非 HAL 协议层面）。

### 7.2 冗余模型的额外启示

MACS-K 的冗余设计与 ECS-700 的差异点对 AUDESYS 有额外启示：

| MACS-K 设计 | AUDESYS 启示 |
|-------------|-------------|
| 三电源隔离（系统/现场/辅助） | Signal 原语的 QoS 可扩展**电源域**（power_domain）属性，标识信号属于哪个电源域 |
| TC/RTD 冗余（行业首创） | 模拟信号（F32）的物理冗余需要考虑传感器冗余场景（如三取二表决） |
| HollySys Safety Switch（DCS 数据包白名单） | StreamChannel 的**数据包过滤**可以作为 HalQoS 的一个维度 |
| 二乘二取二铁路安全架构 | amw 的 Security Domain 可扩展支持 SIL4 级别的安全通信（远期目标） |
| 主从热备（< 3 IEC 周期）vs ECS-700 的 < 1 扫描周期 | 冗余切换时间的差异取决于具体实现，HAL 协议层不应硬编码切换时间 |

### 7.3 跨行业平台策略对 AUDESYS 的启示

和利时"同一平台服务四行业"的策略对 AUDESYS 的设计哲学有深刻启发：

| 和利时策略 | AUDESYS 启示 |
|-----------|-------------|
| 核心安全计算机平台跨行业复用 | AUDESYS Runtime 核心应保持行业无关性，行业特性通过"配置+库"注入 |
| 不同行业的不同安全等级（SIL3/SIL4） | HalQoS 的 Security Domain 应支持多级安全等级配置 |
| 工程工具链跨行业共享 | Studio IDE 应是行业无关的通用平台，行业模板作为插件加载 |
| 供应链跨行业规模经济 | 不绑定特定行业硬件——这是 AUDESYS 作为"仿真平台"的天然优势 |

**核心理念**：AUDESYS 应设计为**通用工业控制仿真平台**，而非针对特定行业的仿真器。行业差异化通过"行业库（Industry Library）"而非核心代码实现。

### 7.4 安全通信模型的深度分析

和利时的铁路 SIL4 和核电安全系统为 AUDESYS 的 Security Domain 设计提供了极端场景参考：

| 安全等级 | 危险故障概率/h | AUDESYS Security Domain | 典型通信要求 |
|---------|---------------|------------------------|-------------|
| SIL1 | < 10⁻⁶ | security_domain: "standard" | 标准 TCP/IP |
| SIL2 | < 10⁻⁷ | security_domain: "enhanced" | CRC 校验 + 序列号 |
| SIL3 | < 10⁻⁷ | security_domain: "safety" | 安全通信层（如 PROFIsafe） |
| SIL4 | < 10⁻⁹ | security_domain: "critical" | 二取二/三取二冗余 + 故障安全 |

**AUDESYS Phase 1 策略**：在 HAL 层面只需支持 security_domain 标记（如 "standard" 和 "safety" 两级），具体的安全通信协议实现（如 PROFIsafe, CIP Safety）放在 amw_transport 实现层。这是 D16 决策的正确方向。

### 7.5 虚拟工厂仿真对 Simulator 的具体启示

MACS-K 的虚拟工厂仿真是对 AUDESYS Simulator 最有价值的对标功能：

| MACS-K 虚拟仿真 | AUDESYS Simulator 兑现方式 |
|----------------|--------------------------|
| 一台 PC 完成全流程仿真 | Simulator 单机模式 |
| 完全等同真实控制逻辑 | Runtime 运行与实际 Runtime 完全相同的控制代码 |
| 与组态工程兼容 | 仿真工程 = 生产工程（同一文件格式） |
| 离线仿真 | Simulator 的加速模式（Fast-Forward） |
| 大幅降低测试成本 | TDD 工作流：先在 Simulator 中测试，再部署 |

**AUDESYS 的核心价值**：为工程师提供"零硬件成本"的控制策略开发和验证环境。

### 7.6 铁路信号 Fail-Safe 哲学对 HAL 的启示

铁路信号的"故障-安全"原则对 AUDESYS HAL 的故障处理设计有参考价值：

1. **任何故障导向安全侧**：HAL 的 Signal 原语在通信故障时应返回**预定义的 Fail-Safe Value**，而非上一周期值或默认零值
2. **输出必须可验证**：RPC 的输出指令应支持"回读验证"（Read-back Verification）
3. **双通道比较**：SIL4 级别的安全输出需要两个独立通道的计算结果一致才输出
4. **看门狗机制**：每个 RT 周期结束后，需检查所有安全关键 Signal 是否在预期范围内更新

虽然 AUDESYS Phase 1 不追求 SIL 认证，但在 Signal 和 RPC 原语中预留 Fail-Safe 回调机制，可以避免后期需要安全认证时的架构重构。

### 7.7 OCS 光总线对 HAL 物理层抽象的启示

和利时 OCS 的"软件定义 I/O"理念提示 HAL 的 Signal 原语应考虑：

- **I/O 通道类型由软件定义**：同一物理通道可以配置为 AI/AO/DI/DO/PI，取决于工程组态
- **物理层透明**：HAL Signal 不应关心底层是铜缆还是光纤
- **信号路径可追踪**：从现场设备到控制逻辑的完整信号路径应可图形化展示（对 Studio IDE 的要求）

### 7.8 HOLLiAS Bridge 对 amw 互操作性的启示

HOLLiAS Bridge 作为"通用互联平台"提示 AUDESYS 的 amw 层需要：

1. **协议适配器模式**：amw_transport 可以设计为适配器模式（Adapter Pattern），一个 transport 实例封装一个协议栈（Profibus-DP/Modbus/HART/OPC UA）
2. **统一数据模型**：不同协议的数据在进入 HAL 之前应归一化为 AUDESYS 的 14 种统一类型（D12 决策）
3. **地址映射**：HAL 的 Pin 命名（component.interface.name）可以映射到 Profibus-DP 的槽/索引地址

### 7.9 不推荐直接复制的部分

| MACS-K 设计 | 不推荐原因 |
|-------------|-----------|
| IO-BUS 直接绑定 Profibus-DP 协议 | AUDESYS 应保持协议无关，通过 amw_transport 适配 |
| INNER-BUS 三层架构（控制器-交换机-IO-BUS 模块） | 额外的网络层增加了复杂性和故障点 |
| 二乘二取二铁路安全架构 | 过于复杂，AUDESYS 仿真场景不需要 |
| 三电源隔离的供电复杂性 | 仿真平台直接由宿主机供电，无需电源隔离设计 |
| 专有的 HollySys Safety Switch 硬件 | AUDESYS 不应绑定特定硬件 |

### 7.10 专用控制器 DEH/ETS 对 Runtime 模块化的启示

和利时在火电领域的 DEH（汽轮机控制）和 ETS（紧急跳闸）作为独立于通用 DCS 的专用控制器，提示 AUDESYS Runtime 的模块化设计：

- Runtime 核心：通用控制执行引擎（支持标准 IEC 61131-3 逻辑）
- Runtime 扩展：行业专用模块（如 DEH 转速控制模块、ETS 跳闸逻辑模块）
- 扩展方式：通过 HAL 的 RPC 原语调用，而非直接耦合

### 7.12 从和利时私有化看工业软件行业趋势

和利时的私有化（2024 年 7 月）反映了中国工业自动化领域的特殊背景：

1. 去美化/自主可控加速：在中美科技竞争背景下，关键基础设施控制系统国产化成为国家安全要求
2. 资本市场估值差异：纳斯达克对中概股的估值可能低于预期，私有化可以摆脱季度报告压力
3. 长期研发投入需要私有化环境：工业控制系统的研发周期长（5-10 年），公开市场短期业绩压力不利于长期投入
4. 与中控的技术路线分化：中控选择 AI+云化（UCS/TPT），和利时选择安全合规+跨行业深耕

对 AUDESYS 的启示：
- 工业软件/仿真平台的发展需要长期主义
- 开源模式可以解决"长期投入 vs 短期回报"的矛盾
- AUDESYS 应避免过度依赖单一资本模式

### 7.13 铁路信号场景的 HAL 建模探索

如果 AUDESYS 未来扩展到铁路信号仿真，HAL 需要以下扩展：

| 铁路信号场景 | HAL 对应 | 挑战 |
|-------------|---------|------|
| 列控中心→有源应答器→车载 ATP | Signal（周期性位置+限速信息传递） | 需要支持移动订阅者（列车位置变化） |
| RBC（无线闭塞中心）→ GSM-R → 车载 ATP | RPC（移动授权请求/响应） | 高延迟、高可靠性，需要超时重传机制 |
| 临时限速服务器→列控中心 | RPC（限速指令下发+确认） | 需要操作审计和数字签名 |
| 联锁→信号机/道岔 | Signal（状态周期性刷新） | 需要 Fail-Safe 值（故障时引导至安全状态） |

虽然铁路信号仿真暂不在 AUDESYS Phase 1 范围内，但 HAL 原语的设计已经可以覆盖这些场景（Signal/RPC 的组合），不需要引入第 4 种原语。这验证了 D10 决策的正确性。
