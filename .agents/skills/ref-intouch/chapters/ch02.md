# Ch02: 技术特性 — Technical Architecture

## 2.1 核心架构五层模型

AVEVA System Platform 的架构可以归纳为五层模型：

```
┌──────────────────────────────────────────────────────────────┐
│                    可视化层（Visualization）                     │
│  InTouch HMI (WindowViewer) / InTouch OMI (HTML5) / Access   │
│  Anywhere (RDP/Web)                                           │
├──────────────────────────────────────────────────────────────┤
│                    应用服务层（Application Server）              │
│  ArchestrA IDE + Application Engines (报警/历史/脚本/I/O)      │
├──────────────────────────────────────────────────────────────┤
│                    配置管理层（Galaxy Repository）              │
│  Galaxy Repository (GR Node) — Microsoft SQL Server           │
│  统一命名空间 / 对象/模板/图形符号/安全配置                      │
├──────────────────────────────────────────────────────────────┤
│                    通信中间件层（Communications）                │
│  SuiteLink / DDE/FastDDE / OPC DA/UA / MQTT / REST / ODBC   │
├──────────────────────────────────────────────────────────────┤
│                    现场设备层（Field Devices）                   │
│  PLC / RTU / DCS / 智能仪表 / 条码扫描器 / RFID               │
└──────────────────────────────────────────────────────────────┘
```

## 2.2 ArchestrA 面向对象自动化框架

ArchestrA 是 System Platform 的技术核心，解决了传统 Tag 模型的三大根本性问题：

**问题 1：平面 Tag 爆炸** — 一个水泵需要 20+ 个 Tag，10 个水泵 = 200+ 个独立 Tag。
**问题 2：工程不可复用** — Tag 模型不支持模板化和继承。
**问题 3：部署与运维割裂** — Tag 配置存在于每台 View Node 的本地文件中。

### ArchestrA 关键概念

| 概念 | 说明 | 类比 |
|------|------|------|
| **Template（模板）** | 定义一类设备的属性、脚本、图形的蓝图 | Class |
| **Derived Template（派生模板）** | 基于父模板的扩展/定制 | Subclass |
| **Instance（实例）** | 模板的具体运行期对象，映射到实际 I/O 地址 | Object Instance |
| **Containment（包容）** | 对象包含子对象形成层次结构 | Composition |
| **Area（区域）** | 对象的组织分组 | — |
| **Deployment（部署）** | 将对象实例指定到特定节点的特定 AppEngine | — |
| **Galaxy（星系）** | 所有配置的统一逻辑命名空间 | Namespace |

## 2.3 Galaxy Repository — 统一配置数据库

Galaxy 基于 Microsoft SQL Server，是 ArchestrA 对象模型的物理载体。

**关键设计原则**：
1. **单一数据源（Single Source of Truth）**：所有配置数据来自一个 SQL Server 数据库。
2. **多用户开发**：多工程师同时开发同一 Galaxy，通过检出/检入管理冲突。
3. **集中部署**：从 IDE 中选择"Deploy"，对象被推送到指定 AppEngine 节点。
4. **命名空间隔离**：所有对象引用使用 Galaxy 级别的名称。

> 复杂度警告：Galaxy 的管理开销显著。对于仅需单机 HMI 的小项目，Galaxy 的管理开销可能比其带来的好处更大。

## 2.4 WindowMaker 与 WindowViewer — IDE/Runtime 分离设计

这是 InTouch 最具影响力的设计决策之一。

### InTouch 应用类型

| 应用类型 | 创建方式 | 管理方式 |
|----------|---------|---------|
| **Standalone Application** | Application Manager | Application Manager |
| **Managed Application** | ArchestrA IDE | ArchestrA IDE |
| **InTouchView Application** | Application Manager 或 IDE | Application Manager |

**快速切换（Fast Switch）**：WindowMaker 和 WindowViewer 之间支持快速切换，编辑后立即在 Runtime 中测试。

## 2.5 Tag 数据模型

### Tag 类型体系

| 分类 | 类型 | 说明 |
|------|------|------|
| **Memory Tag** | Discrete/Integer/Real/Message | 内部变量，WindowViewer 内存驻留 |
| **I/O Tag** | I/O Discrete/Integer/Real/Message | 外部数据源（PLC/RTU/远程节点） |
| **特殊 Tag** | Indirect Tag / Hist Trend Tag / Tag ID Tag / SuperTag | 标签指针、趋势引用、模板 |
| **系统 Tag** | $System Tag（100+ 个预定义） | 系统预定义，$ 前缀标识 |

### Tag 的进化：Tag-based → Object-based

| 阶段 | 模型 | 局限 |
|------|------|------|
| **Tag-based**（InTouch 1.0-9.x） | 全局扁平 Tag 列表 | Tag 爆炸、无层次结构 |
| **SuperTag**（InTouch 7.0+） | Tag 模板（关联 Tag 组） | 无行为封装、无继承 |
| **ArchestrA Object**（System Platform 1.0+） | 面向对象（属性+脚本+图形） | 学习曲线陡峭 |
| **InTouch Unlimited**（2024+） | 混合模型 | 构建中 |

## 2.6 QuickScript 脚本引擎

七种脚本类型（按触发器分类）：

| 脚本类型 | 触发器 | 典型用途 |
|----------|--------|----------|
| **Application Script** | On Startup / While Running / On Shutdown | 初始化变量、后台循环 |
| **Window Script** | On Show / While Showing / On Hide | 窗口级逻辑 |
| **Key Script** | On Key Down / While Down / On Key Up | 键盘快捷键 |
| **Condition Script** | On True / While True / On False / While False | 基于 Tag 值的条件逻辑 |
| **Data Change Script** | Tag 值变化（仅一次） | 报警处理、事件响应 |
| **Action Script** | 操作员点击图形对象 | 按钮动作、命令下发 |
| **ActiveX Event Script** | ActiveX 控件事件 | 第三方控件集成 |

## 2.7 通信架构 — I/O Server 模型

### 通信协议栈

| 协议 | 传输层 | 当前状态 |
|------|--------|----------|
| **SuiteLink** | TCP/IP（VTQ） | 推荐（工业级 TCP 通信） |
| **DDE/FastDDE** | Windows 消息 | 已过时 |
| **OPC DA** | COM/DCOM | 成熟稳定 |
| **OPC UA** | TCP/HTTPS | 当前主流 |
| **MQTT** | TCP | 新增（2023+） |
| **Access Name** | 抽象层 | **核心设计** |

**Access Name 机制**：每个 Access Name 定义了 InTouch 到 I/O Server 的一条逻辑连接，在 WindowMaker 中配置 I/O Tag 时只需指定 Access Name 和 Item Name，无需了解底层协议细节。

## 2.8 InTouch OMI（Operations Management Interface）

AVEVA 对现代 Web 可视化的答案：

| 特性 | 说明 |
|------|------|
| **HTML5 原生** | 现代浏览器运行，无需 ActiveX 或插件 |
| **响应式设计** | 一次构建，自适应桌面/平板/手机/大屏 |
| **情境感知 UI** | 根据用户角色、设备类型、位置自动调整 |
| **App 架构** | 卡片式功能单元（趋势、报警、地图、KPI） |
| **动态自动生成** | 根据模板定义自动生成画面 |
| **Situational Awareness** | 基于 ISA-101 标准的色彩方案和导航模式 |

## 2.9 分布式架构

### 分布式关键概念

| 概念 | 说明 |
|------|------|
| **WinPlatform** | 物理计算机在 Galaxy 中的代表对象 |
| **AppEngine** | 运行 Automation Object 的后台服务 |
| **ViewEngine** | 可视化引擎后台服务 |
| **Area** | 对象组织分组，决定运行在哪个 AppEngine |
| **冗余** | AppEngine Primary/Backup 对，自动故障转移（约 15 秒） |

### 部署模型建议

| 应用规模 | I/O 点数 | 推荐架构 |
|----------|---------|---------|
| 小型 | < 25,000 | 单节点 All-in-One |
| 中型 | 50,000-200,000 | Client-Server + 冗余 AppEngine |
| 大型 | > 200,000 | 多 AppEngine + 分布式 GR + RDS |
