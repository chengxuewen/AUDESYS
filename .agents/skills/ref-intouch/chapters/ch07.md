# Ch07: 对 AUDESYS 参考价值 — Reference Value

## 7.1 ArchestrA 对象模型 → HAL Device Model 设计

| ArchestrA 概念 | AUDESYS 对应 | 参考价值 |
|---------------|-------------|---------|
| **Object Template** | HAL Device Model 模板 | 定义设备的 Signal/StreamChannel/RPC 接口套餐 |
| **Instance** | Device Instance | 运行时实例化，绑定到物理 I/O |
| **Containment** | Device Tree | 模块包含子模块 |
| **Derived Template** | 设备模型继承 | 基类设备 → 派生设备 |
| **Deployment** | Task/Thread Assignment | 分配到特定 RT 线程 |
| **Attribute** | Signal / StreamChannel | 带时间戳和质量的信号 |

### AUDESYS 可以借鉴的设计模式

```
DeviceTemplate MotorBase:
  Signals:
    Running:     Bool    (output, write-only)
    Speed:       Float64 (output, write-only)
    Current:     Float64 (input, read-only)
    Temperature: Float64 (input, read-only)
    FaultCode:   Uint16  (input, read-only)
  StreamChannels:
    VelocityProfile: StreamChannel (output, history=1000)
  RPC:
    Enable():    void
    Disable():   void
    ResetFault(): void
    HomeAxis():  void

DeviceTemplate ServoMotor extends MotorBase:
  Signals:
    + Position:   Float64 (input, read-only)
    + Torque:     Float64 (input, read-only)
    + FollowingError: Float64 (input, read-only)
```

## 7.2 Galaxy 集中式配置 → Studio IDE 配置管理

| Galaxy 特性 | AUDESYS 参考 | 建议 |
|------------|-------------|------|
| **单一数据源** | Studio Project 文件 | YAML/JSON 或 SQLite 统一项目文件 |
| **多工程师协作** | Git-based 版本控制 | Studio 集成 Git，支持分支/合并 |
| **逻辑命名空间** | Device Tree + Signal 命名 | `component.interface.name` 命名 |
| **部署工作流** | Config Barrier（D17） | 配置变更排队到周期边界 |
| **模板-实例** | Device Template Library | Studio 提供设备模板库 |
| **对象版本管理** | Git tags + semver | 设备模板版本管理 |

**关键教训 — Galaxy 的复杂度代价**：过度集中化带来运维负担。AUDESYS 应采用轻量级的 Git-based 配置管理（Configuration as Code）。

## 7.3 WindowMaker ↔ WindowViewer — Studio ↔ Runtime 对标

| InTouch | AUDESYS | 分析 |
|---------|---------|------|
| **WindowMaker** | Studio IDE | 图形化工程环境 |
| **WindowViewer** | Runtime Engine | 实时执行环境 |
| **Fast Switch** | Hot Reload | 编辑后立即在 Runtime 中测试 |
| **NAD** | Runtime 远程部署 | 从 Studio 部署到 Runtime 节点 |
| **Managed Application** | Studio-Managed Project | IDE 全生命周期管理 |
| **Tagname Dictionary** | Signal/StreamChannel Registry | 数据点定义和管理 |

**需注意的设计差异**：

| 维度 | InTouch | AUDESYS 建议 |
|------|---------|-------------|
| 数据类型 | 4 种 | 14 种（D12 决策：IEC 61131-3 全覆盖） |
| 数据模型 | 平面 Tag → 对象 | 原生 Signal/StreamChannel/RPC |
| 脚本触发 | 7 种触发器 | HAL 层不需要脚本 |
| 通信协议 | SuiteLink/DDE/OPC 混合 | amw 三极 trait 统一抽象 |

## 7.4 Tag-based → Object-based 演进 — 历史教训

**教训 1：不要从平面 Tag 开始**
- 100 个泵 = 2000+ 个独立 Tag，报警配置需手工复制
- "泵类型"不存在于数据模型中
- 跨项目复用为零
- AUDESYS 从第一天就应采用 Signal（component.interface.name）的层次命名模型

**教训 2：向后兼容的代价巨大**
- ArchestrA 必须同时支持 Standalone 和 Managed 两种模式
- 两种脚本语法、两种 Tag 访问方式至今增加维护负担
- AUDESYS 的 HAL 只有一种数据模型：Signal/StreamChannel/RPC

**教训 3：对象模型 + 图形符号绑定**
- ArchestrA 的最佳设计选择之一是将对象模板与 Industrial Graphics 绑定
- AUDESYS Studio 应借鉴：拖入设备模板到画面中，自动生成标准控件

## 7.10 需警惕的陷阱

1. **Tag 模型的技术债**：AUDESYS D10 已确立三分法是正确的。不要引入平面 Tag 兼容层。
2. **双重产品线维护负担**：InTouch OMI 与 WindowViewer 双线并行造成长期负担。AUDESYS Studio 应选定唯一技术栈。
3. **许可定价的透明性**：InTouch"不公开标价"造成市场不信任。
4. **过度工程化**：Galaxy + ArchestrA 管理复杂性对小项目是负担。AUDESYS 应保持"渐进式复杂度"。
5. **向后兼容的代价**：InTouch 至今支持 DDE、NetDDE、Group Var Tag 等遗留产物。
6. **Windows 绑定**：对 COM/DCOM、ActiveX 的深度依赖限制了平台选择。AUDESYS Rust/C 实现可避免此问题。
7. **安装体积膨胀**："仅装 InTouch 也必须下载 4.5GB"增加部署摩擦。
