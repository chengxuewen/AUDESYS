## 7.2 IDE 设计参考

**iFIX WorkSpace 优点**：
- 配置与运行在同一环境
- 拖拽式开发，降低编程门槛
- 丰富的预置对象（Dynamo）
- 脚本扩展（VBA）提供灵活性

**AUDESYS 可借鉴**：
- "配置为主 + 脚本扩展"的开发模式
- 内置丰富的工业符号库
- 统一的项目浏览器（树形结构）
- 在线调试和变量监控

**需避免的缺陷**：
- VBA 脚本的局限性（性能、安全性）→ AUDESYS 选择 WASM + Python（D26）
- 配置工具分散 → AUDESYS 统一 IDE
- 版本兼容性管理复杂 → FlatBuffers schema 版本管理

## 7.3 通信协议抽象层

KEPServerEX/IGS → amw（AUDESYS Middleware）类比：

- **统一 OPC 层** → HAL 统一通信接口
- **驱动插件化** → amw_inproc / amw_zenoh 可替换
- **协议转换** → HAL 原生协议设计（禁止桥接外部协议，D10）

## 7.4 可视化与 HMI 设计

iFIX High Performance HMI 对 AUDESYS Studio 的影响：

- 异常驱动显示 → Studio HMI 设计器原则
- 减少视觉噪音 → 颜色语义化策略
- 层级导航 → 画面导航结构设计
- Dynamo 对象 → 可视化模板库

## 7.5 报警与事件系统

iFIX 报警系统是工业 SCADA 标杆：

- ISA 18.2 标准合规
- 报警分区和路由
- 优先级和升级机制
- 操作员确认流程

**AUDESYS 参考**：
- Runtime 内置报警系统设计
- 报警模型与 HAL Signal/StreamChannel 原语集成
- 报警存储和查询策略（Proficy Historian 模式）

## 7.6 所有权变迁的教训

Intellution → Emerson → GE → GE Vernova → Velotic

**教训**：
1. 品牌多次变更（FIX → FIX32 → iFIX → Proficy → GE Vernova → Velotic）
2. 每次所有权变更引发用户担忧
3. 技术债务（VBA/COM 老旧，但向后兼容限制现代化）
4. 独立 vs 依附的权衡

**对 AUDESYS 的参考**：
- 保持品牌一致性
- 核心技术栈考虑长期演进
- 独立运营的商业模式设计
- 技术现代化与向后兼容的平衡

## 7.8 对 AUDESYS Studio IDE 的具体建议

1. **统一 IDE 设计**: 效仿 WorkSpace 的"配置即运行"，但用现代技术栈
2. **插件化架构**: 参考 CODESYS 插件系统 + iFIX ActiveX 集成
3. **脚本引擎**: 选择现代语言（WASM/Python）替代 VBA（D26）
4. **可视化内置**: 内置 HMI 设计器，High Performance HMI 原则
5. **通信抽象层**: HAL amw 统一协议接入
6. **Web 支持**: 原生 Web 部署（借鉴 Operations Hub / WebVisu）
7. **开放标准**: OPC UA, MQTT, REST API
