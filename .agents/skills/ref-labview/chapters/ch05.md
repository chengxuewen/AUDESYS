# Chapter 5: Hardware Abstraction (DAQmx/VISA)

## 5.1 NI-DAQmx Architecture

DAQmx is NI's unified data acquisition driver API. It is the single most successful commercial HAL implementation in the test/measurement industry.

### Channel Model

```
Physical Channel
  -> Virtual Channel
    -> DAQmx Task (virtual channel + timing + trigger config)
      -> Read/Write VI (actual I/O operation)
```

**Physical Channel**: Actual device I/O terminal (e.g., "Dev1/ai0" = device 1 analog input channel 0)

**Virtual Channel**: Software representation of physical channel, with custom name, scaling, and engineering unit mapping. The name is defined in config, not hardcoded.

**Task**: Complete acquisition/output configuration (channel selection, sample rate, sample mode, trigger conditions, clock source). An atomic operation unit.

**DAQmx Read/Write VI**: Executes actual data acquisition or signal output. Supports N-sample, single-point, continuous acquisition modes.

### Channel Abstraction Comparison

| DAQmx Layer | AUDESYS HAL Equivalent | Analysis |
|------------|------------------------|----------|
| Task | Signal (configured I/O abstraction) | Both encapsulate configuration + execution |
| Virtual Channel | component.interface.name | Both provide logical names independent of physical addressing |
| Scaling/Engineering Units | Type system conversion layer | DAQmx config in GUI, AUDESYS via type system |
| Trigger Config | Config Barrier config-plane management | DAQmx Task-level, AUDESYS cycle-boundary batch |
| Read/Write VI | Signal read/write direction | DAQmx VI can be sync or async, Signal direction defined at compile time |

### DAQmx Key Design Features

**Unified API**: Same API functions regardless of underlying hardware (USB, cDAQ, PXI). This is DAQmx's most important design decision — identical programming experience across the entire hardware line.

**Task Lifecycle**: 
1. Create Task (configure)
2. Start Task (execute)
3. Read/Write within loop
4. Stop and Clear Task (cleanup)

Three-phase lifecycle maps to AUDESYS HAL's anticipated lifecycle pattern: Signal declaration in Config, operation at Runtime, teardown at shutdown.

**Auto Resource Management**: DAQmx handles device initialization, configuration, acquisition, and cleanup. User does not need to manually manage device open/close (unlike VISA sessions).

**Multithreaded Safety**: DAQmx supports parallel acquisition from multiple tasks on the same device. Device driver manages concurrent access.

**DAQ Assistant**: Graphical configuration wizard generates DAQmx code automatically. Lowers entry barrier for non-programmers. AUDESYS Studio could offer equivalent "Signal wizard" for declarative config generation.

## 5.2 NI-VISA Architecture

VISA (Virtual Instrument Software Architecture) provides unified instrument control:

- **Resource Manager**: Auto-discovers and enumerates available instruments
- **Session Management**: Each connection is a session with session handle
- **Unified I/O**: GPIB/Serial/USB/Ethernet same API (viWrite/viRead)
- **Sync/Async I/O**: Supports both blocking and non-blocking operations

### VISA Design for AUDESYS

VISA's session model (open session -> I/O -> close session) maps to AUDESYS amw_transport:
- VISA Resource Manager -> HalDiscovery trait (enumerate available transports)
- VISA Session -> transport session handle
- viWrite/viRead -> Signal write/read operations

**Critical distinction**: VISA is protocol-agnostic (same API for GPIB, serial, Ethernet). AUDESYS amw_transport should be similarly agnostic — same API for Signal/StreamChannel regardless of underlying IPC or network transport.

## 5.3 NI-MAX (Measurement & Automation Explorer)

MAX is NI's hardware configuration and management tool:

- Device discovery and configuration
- Self-test and diagnostic execution
- Software install and version management
- Simulated devices for development (virtual bench)

**AUDESYS equivalent**: HalDiscovery trait + Studio config UI. Unlike MAX (GUI-dependent), AUDESYS must support CLI/YAML-first configuration with GUI as an optional overlay.

## 5.4 DAQmx Timing and Triggering

DAQmx provides sophisticated timing and trigger models that AUDESYS Config Barrier should study:

- **On-demand**: Single-point acquisition triggered by software
- **Hardware-timed**: Free-running acquisition at programmable rate, hardware clock source
- **Digital Trigger**: Start/stop/reference trigger via PFI lines
- **Analog Trigger**: Level-detect trigger on analog channel
- **Sync**: Multi-device synchronization via shared clock and trigger lines

DAQmx configures all timing/trigger parameters on the Task, not on each individual read/write operation. Similarly, AUDESYS Config Barrier should batch all configuration changes at cycle boundaries, ensuring deterministic config application.

## 5.5 Comparison Summary

| Feature | DAQmx | AUDESYS HAL |
|---------|-------|-------------|
| Abstraction granularity | Channel level (Task) | Component interface (component.interface.name) |
| Configuration | GUI wizard + MAX tool | YAML/JSON config + Studio topology view |
| Read/Write semantics | Read/Write VI (sync/async) | Signal direction + StreamChannel buffered stream |
| Timing model | Task-level config | Thread cycle-level schedule (D13) |
| Hardware discovery | MAX tool | HalDiscovery trait (amw middleware) |
| Error handling | Error Cluster (runtime, wire-propagated) | Rust Result<T,E> (compile-time) |
| Type safety | Wire color + compile-time check | Rust type system (compile-time) |
| Multi-vendor support | NI hardware + IVI/VISA limited | amw transport trait (pluggable protocol stack) |
