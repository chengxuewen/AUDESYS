# Chapter 2: Technical Architecture & G Language

## 2.1 G Language Syntax Elements

G (Graphical Language) uses visual elements instead of keywords and punctuation:

1. **Nodes**: Function VIs, SubVIs, Express VIs. Each node has input and output terminals. Nodes consume (read inputs) and produce (generate outputs) data.
2. **Wires**: Colored lines connecting terminals. Color = data type (blue=integer, orange=float, green=boolean, pink=string, purple=path). Thickness = dimension (thin=scalar, thick=array).
3. **Terminals**: Front panel controls/indicators on the block diagram. Controls have bordered icons, indicators have borderless icons.
4. **Structures**: Framework elements controlling execution flow:
   - While Loop / For Loop
   - Case Structure (switch/case)
   - Event Structure (event-driven GUI)
   - Sequence Structure (forced sequential execution)
   - Formula Node (C-like math expressions within diagram)
   - MathScript Node (.m file script execution)
   - Timed Loop (precise cycle timing)
   - Diagram Disable Structure (compile-time conditional exclusion)
5. **Shift Register**: Nodes on loop boundaries. Left output = previous iteration value, right input = current iteration value. Core mechanism for maintaining loop state.
6. **Feedback Node**: Single-line feedback loop for iterative algorithms (IIR filters).
7. **Property Node / Invoke Node**: Programmatic access to control/VI properties and methods.

## 2.2 Type System

LabVIEW's type system is visually encoded:

| Wire Color | Type | Examples |
|-----------|------|----------|
| Blue | Integer | I8, I16, I32, U8, U16, U32 |
| Orange | Float | Single, Double, Extended precision |
| Green | Boolean | True/False |
| Pink | String | Text, path |
| Purple | Variant/Path | Dynamic data, file paths |
| Brown | Cluster | Compound data structure (struct-like) |
| Array thick wire | Any[] | Thick variant of base color |

Types propagate forward along wires. Strict compile-time type checking — mismatched types produce a "broken wire" error that prevents execution.

## 2.3 Dataflow Execution Semantics

LabVIEW's dataflow model in detail:

- **Data-driven firing**: A node fires (executes) when ALL input terminals have data available. No data, no execution.
- **Token passing**: Think of each wire carrying a "data token". When the token arrives at all input terminals, the node consumes them, executes, and produces output tokens.
- **Implicit parallelism**: If node A and node B share no data dependency, both execute in parallel. The runtime engine manages thread pool allocation automatically.
- **No side effects (pure functions)**: Standard VIs depend only on inputs and produce only outputs. No global mutable state (unless explicitly using Functional Global Variables or LVOOP class data).
- **Deterministic subset**: On RT targets, Timed Loops provide cycle-level determinism with bounded jitter.

**Key difference from control flow**: In text languages, `a = f1(x); b = f2(x)` executes f1 then f2 sequentially even if they have no dependency. In dataflow, f1 and f2 execute in parallel because both only depend on x.

## 2.4 Compilation Pipeline

LabVIEW's compilation process differs fundamentally from text language compilers:

1. User edits block diagram, clicks Run button
2. Compiler parses diagram into DFIR (DataFlow Intermediate Representation)
3. Type propagation: from control/indicator types, derive all node input/output types along wires
4. Dead code elimination: remove code paths not connected to any output terminal or indicator
5. Common subexpression elimination
6. LLVM IR generation (LabVIEW 2017+ switched from proprietary code generator to LLVM backend)
7. Machine code generation, loaded into execution engine

**Edit-Compile-Run Loop**:
- Small VI: 0.1-2 seconds, imperceptible
- Large project (thousands of VIs): first compile minutes, incremental compile on modified VIs only
- Incremental granularity: VI-level (LabVIEW) vs crate-level (Rust)

**Key tradeoffs**:
- Compile speed > runtime performance (LabVIEW prioritizes 1-second Run button response)
- LLVM backend improved numerical computation performance but increased compile time
- FPGA compilation is entirely different: G -> VHDL -> Xilinx/Intel synthesis (minutes to hours)

## 2.5 Execution Engine

- **JIT-like behavior**: Recompiles on every Run click during development. AOT compilation when building EXE.
- **Thread pool**: Execution system maintains a thread pool, automatically assigns parallel nodes to threads.
- **Deterministic execution**: On RT targets, no dynamic memory allocation, no GC, preallocated loop structures.
- **Cross-platform binary**: Compiled .vi files are binary (not human-readable intermediate code). .rtexe for real-time targets.
- **Type propagation**: Enforced at compile time. Broken wire = cannot run.

## 2.6 G Web Development Software

G Web extends LabVIEW to browser-based deployment:

- **WebVI**: LabVIEW-like graphical programming compiled to WebAssembly + JavaScript. Runs in browser.
- **Web HMI**: Browser-accessible test/control interfaces, no LabVIEW runtime installation needed.
- **NI Web Server**: Hosts WebVIs and REST APIs.
- **Limitations**: WebVI functionality far below classic LabVIEW (lacks most toolkits and hardware drivers). Suitable for HMI/SCADA web deployment, not core control logic.

## 2.7 Cross-Language Interoperability

LabVIEW provides multiple interoperability channels:

- **Python Node**: Call Python scripts with bidirectional data transfer
- **.NET Interop**: Constructor/Property/Invoke nodes for .NET objects
- **DLL Call Library**: Call C/C++ shared libraries with configurable calling convention
- **ActiveX/COM**: Windows COM component integration
- **gRPC**: Service-based communication (added in 2022)
- **OPC UA**: Industrial connectivity
- **MQTT/HTTP/WebSocket**: Network protocol support
