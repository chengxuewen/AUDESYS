# Chapter 7: AUDESYS Mapping & Design Implications

This chapter maps LabVIEW concepts directly to AUDESYS design decisions. Use this as the primary reference when making AUDESYS architecture or implementation decisions.

## 7.1 Dataflow Communication Primitives Mapping

LabVIEW's 30+ year evolution validates AUDESYS D10 (three-primitive separation):

| LabVIEW Concept | AUDESYS HAL Primitive | Analysis |
|----------------|----------------------|----------|
| Wire (connection between nodes) | Signal (single-writer, multiple-reader, latest-value) | Wire passes data from source to target — data arrives, target fires. Same semantics. HAL Signal can be async (cross-process), LabVIEW wire is synchronous. |
| Queue / Channel Wire | StreamChannel (multiple-writer, multiple-reader, buffered) | Channel Wire (LabVIEW 2016) explicitly distinguishes Stream Channel (buffered, like Kafka) vs Tag Channel (latest-value, like Signal). Same separation AUDESYS chose independently. |
| VI Call (SubVI invocation) | RPC (request/response, requires acknowledgment) | Synchronous SubVI call = sync RPC. Async Call = async RPC. |
| Network-Published Shared Variable | Signal + HalDiscovery | Multi-node data sharing with auto-discovery. |
| Notifier (value-change notification) | Signal subscription notification | Notifier wakes all waiting receivers on value change. |

**Key insight**: LabVIEW's 2016 Channel Wire introduced the same Signal-vs-StreamChannel distinction that AUDESYS D10 codified. This is independent convergent design validation — two different engineering teams arrived at the same conclusion about the two irreducible communication patterns.

## 7.2 DAQmx Channel Model to HAL Naming

DAQmx's three-layer abstraction directly maps to AUDESYS HAL naming:

```
DAQmx:  Physical Channel -> Virtual Channel -> Task (configured I/O)
HAL:    Hardware address   -> component.interface.name -> Signal (configured I/O)
```

**DAQmx design principles AUDESYS should adopt**:
- Physical addressing defined in config files, not in code
- Logical names decoupled from physical channels
- Task encapsulates channel + timing + trigger as an atomic unit

**DAQmx design principles AUDESYS should improve**:
- Configuration should be CLI/YAML-first, not GUI-dependent (avoid MAX dependency)
- Type safety should be compile-time (Rust type system), not wire-color visual

## 7.3 RT Scheduling Validation

LabVIEW RT confirms D13 four-system hybrid thread scheduling:

| LabVIEW RT Feature | AUDESYS D13 Mapping |
|-------------------|--------------------|
| Timed Loop deterministic cycle | RT thread (LinuxCNC explicit function list) |
| CPU core isolation (affinity) | RT thread exclusive CPU core |
| Watchdog timer timeout detection | HalQoS deadline monitoring (D16) |
| Multi-rate Timed Loops | Multi-cycle RT thread groups |
| Host VI / Target VI separation | Studio/HMI -> Runtime separation |
| Network Streams (host <-> RT) | amw_zenoh transport (cross-node Signal/StreamChannel) |
| Non-RT loops for logging/diagnostic | I/O threads + event-driven threads |

**Validation conclusions**:
- Deterministic RT loops can coexist with HTTP servers and loggers on the same hardware (core isolation + priority scheduling)
- "Hard real-time" and "soft/non-real-time" task coexistence is feasible
- Cross-host communication (host <-> RT target) at 100us-1ms latency is acceptable (matches amw_zenoh ~100us target)

## 7.4 Studio IDE Design Implications

### LabVIEW features to adopt for AUDESYS Studio

| LabVIEW IDE Feature | AUDESYS Studio Application |
|--------------------|---------------------------|
| Front Panel = UI / Block Diagram = Logic | Dual-view mode: HMI layout panel + control logic panel |
| Highlight Execution (data flow animation) | Debug bridge: highlight Signal/StreamChannel data paths |
| Probe Tool (hover-to-read live values) | Debug panel: real-time Signal values and StreamChannel buffer status |
| Instant compile-and-run (edit-click-run loop) | Fast development-test-debug cycle. Incremental ST compilation <1s |
| Express VI (config wizard generates VI) | Config wizards for standard control loops (PID parameterization, alarm rules) |
| Color-coded types (blue=int, orange=float) | Visual type system with color cues |

### LabVIEW anti-patterns to avoid

| Anti-Pattern | AUDESYS Avoidance |
|-------------|-------------------|
| Block diagram spaghetti in large projects | ST as primary editing mode. Graphical as auxiliary overview/debug tool |
| Binary .vi file format (not Git-diffable) | Text-based project files (JSON/YAML/TOML). Git-diffable from day 1. |
| GUI-dependent hardware config (MAX) | CLI + YAML-first HAL config. GUI as optional overlay. |
| Monolithic IDE architecture | Plugin-based IDE architecture (VS Code model). |
| NI hardware lock-in | amw transport trait keeps hardware independent. |

### Recommended Development UX

```
AUDESYS Studio development cycle:
1. Write ST code (primary) or configure visual logic (auxiliary)
2. Save -> auto-trigger incremental compile (<1s feedback)
3. Run in Simulator (no hardware needed)
4. Debug with data-flow visualization (Signal highlighting, probe values)
5. Deploy to Runtime target via one-click
```

## 7.5 Project File Format Design

LabVIEW's binary .vi format is the single biggest lesson for AUDESYS:

**AUDESYS project file principles**:
- Control logic (ST code) -> plain text .st files
- HAL topology config -> YAML files (declarative, Kubernetes-style)
- Signal/StreamChannel connections -> JSON/YAML mapping tables
- Project metadata (version, author, dependencies) -> TOML files (Cargo.toml style)
- HMI layout definition -> JSON (parseable by web frontend renderer)

**Benefits of text-first approach**:
1. Git DIFF shows human-readable change differences
2. Merge conflicts resolved via standard text merge tools
3. Code review directly in GitLab/GitHub PR interface
4. CI/CD pipeline directly reads/modifies project files
5. AI coding assistants (LLMs) can directly understand and generate project files

## 7.6 Compilation Pipeline Implications

LabVIEW's compiler design offers lessons for AUDESYS AST -> HIR -> IR pipeline:

| LabVIEW Compiler Feature | AUDESYS Implication |
|------------------------|--------------------|
| DFIR (DataFlow Intermediate Representation) | HIR as stable interface. Phase 2+. |
| Incremental VI-level compilation | Incremental crate-level compilation. Rust cargo already supports this. |
| Compile speed > runtime performance in dev | Debug mode (fast compile, debug symbols) vs Release mode (full optimization) |
| LLVM backend (2017+) | Rust also uses LLVM. Shared optimization infrastructure. |
| FPGA compilation (minutes to hours) | Not applicable — AUDESYS targets CPU-only. |

**Studio compile UX**:
- Editor save -> auto-trigger LSP incremental analysis
- Compile feedback in <1 second (like LabVIEW's Run button response)
- Debug mode: unoptimized, debug symbols
- Release mode: fully optimized for production deployment

## 7.7 Error Model Contrast

| Feature | LabVIEW Error Cluster | AUDESYS (Rust) Error Handling |
|---------|----------------------|------------------------------|
| Type | Runtime check (error propagation at runtime) | Compile-time check (Result<T,E> enforced) |
| Short-circuit propagation | VI skips execution (implicit) | `?` operator (explicit short-circuit) |
| Error merging | Merge Errors VI (explicit) | Iterator::collect() or custom combinators |
| Sequence forcing | Error Cluster wire forces serial execution | `?` operator defines control flow implicitly |
| User notification | General Error Handler VI (GUI popup) | Logging system + HalQoS alerts |

**AUDESYS should use Rust Result<T,E> because**:
- Compile-time enforcement eliminates entire class of "forgot to check error" bugs
- Rust type system supports precise error types (not I32 error codes + string descriptions)
- `?` operator is cleaner than Error Cluster wiring (no spaghetti on complex diagrams)

## 7.9 Fundamental Paradigm Difference

LabVIEW's dataflow execution model and AUDESYS's IEC 61131-3 cycle-scan model are two fundamentally different paradigms:

| Dimension | LabVIEW Dataflow | AUDESYS IEC 61131-3 Cycle Scan |
|-----------|-----------------|-------------------------------|
| Execution trigger | Data-ready = immediately fire | Fixed cycle scan. All logic executes once per scan cycle. |
| Execution order | Implicit data-dependency order (wire flow) | Task scheduler explicit order (predefined order each cycle) |
| Real-time guarantee | Timed Loop provides determinism. Ordinary While Loop has no timing guarantee. | All control tasks guaranteed deterministic cycle by RT scheduler. |
| Loop semantics | Code in While Loop may parallelize across iterations (if data dependencies allow) | Cycle boundary. All logic completes within current cycle. No cross-cycle parallelism. |
| State management | Shift Register (explicit feedback node on loop boundary) | Function block instance persistent members (function block instances survive across scan cycles). |
| Debug experience | Highlight Execution node-by-node dataflow tracing (intuitive but slow) | Breakpoints + single-step + variable watch (traditional debug experience) |

**AUDESYS implications**:
- Runtime uses cycle-scan model (deterministic, predictable). Not dataflow model (non-deterministic scheduling order).
- Studio debug can borrow "dataflow visualization" — highlight Signal/StreamChannel paths to help engineers understand information flow.
- Studio should NOT implement full graphical dataflow programming (that is LabVIEW's domain). Focus on ST smart assistance and visual debugging.
