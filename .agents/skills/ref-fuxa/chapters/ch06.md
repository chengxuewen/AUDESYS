# Chapter 6: Product Differentiators

## 6.1 SVG Editor — "Web HMI's Figma"

FUXA's SVG editor is its most prominent technical differentiator — a **browser-based industrial graphics editor**:

- **Fully Web-based**: No desktop software installation — open browser and design HMI screens
- **WYSIWYG**: What you see in design is what you see at runtime
- **Visual data binding**: Configure Tag binding through property panel, not by writing code
- **Zero programming threshold**: Automation engineers can create monitoring screens without HTML/CSS/JavaScript knowledge

This design philosophy aligns closely with Ignition Perspective Designer (in-browser HMI designer), but FUXA is completely open-source.

## 6.2 Lightweight Architecture — $35 SCADA Server

FUXA runs on Raspberry Pi (hardware cost only $35), making it one of the **lowest-cost SCADA deployment options**:

```
Minimum deployment hardware cost comparison:
- FUXA on Raspberry Pi 4:       $35 (RPi 4) + $10 (SD card) = ~$45
- Ignition Edge on industrial PC: $1,500 (Edge License) + $500 (IPC) = ~$2,000
- WinCC RT on IPC:              $3,000+ (License) + $1,000 (IPC) = ~$4,000+
- KingView on Windows IPC:      ¥5,000 (License) + ¥3,000 (IPC) = ~¥8,000
```

While FUXA's feature completeness doesn't match commercial products, this cost advantage is disruptive for small projects or prototype validation.

## 6.3 MIT License — True Freedom

MIT License is one of the **most permissive open-source licenses**:

- Allows commercial use (no need to open-source your own code)
- Allows modification and distribution (no need to disclose changes)
- Allows closed-source integration (can embed in commercial products)
- Only requirement: retain copyright notice

This means enterprises can use FUXA as their product's HMI module, even re-selling it closed-source after secondary development.

## 6.4 Docker-Native Support

FUXA is one of the few open-source SCADA projects offering an **official Docker image**:

```bash
# One-line deployment
docker run -d --name fuxa \
  -p 1881:1881 \
  -v fuxa_data:/usr/src/app/FUXA/server/_appdata \
  -v fuxa_db:/usr/src/app/FUXA/server/_db \
  frangoteam/fuxa:latest
```

Docker benefits:
- Environment consistency (no Node.js version or OS differences to consider)
- Quick deployment and migration
- Docker Compose multi-service orchestration (FUXA + InfluxDB + Mosquitto MQTT)
- Kubernetes deployment support (requires self-configured storage volumes)

## 6.5 Node.js Ecosystem — Familiar Development Environment

Choosing Node.js as backend is a strategic technical decision:

- **Huge npm ecosystem**: Direct access to hundreds of thousands of npm packages
- **JavaScript full-stack**: Same language for frontend and backend, lowering team skill requirements
- **Asynchronous I/O advantage**: Node.js event-driven model well-suited for handling high-concurrency device communication
- **Low learning curve**: Web developers can transition to industrial automation development

**Caveat**: Node.js garbage collection (GC) may cause **real-time performance uncertainty** — acceptable for millisecond-level SCADA timing, but unsuitable for microsecond-level deterministic hard real-time control.

## 6.6 Plugin-Based Protocol Adapters

FUXA's device protocols use a **plugin architecture** — adding new protocols is relatively simple:

```
protocols/
  modbus.js      ← Modbus TCP/RTU adapter
  opcua.js       ← OPC UA adapter
  mqtt.js        ← MQTT adapter
  s7.js          ← Siemens S7 adapter
  bacnet.js      ← BACnet adapter
  ethernetip.js  ← EtherNet/IP adapter
  webapi.js      ← WebAPI adapter
```

Each adapter implements a unified interface (connect, disconnect, read, write). Adding a protocol only requires implementing the same interface.

## 6.7 Portability of Engineering Files

FUXA stores entire projects as JSON config files, enabling high portability:

- All SCADA configuration (devices, Tags, screens, alarms, users) lives in JSON files under `_appdata/`
- Projects can be migrated between computers via simple file copy
- Projects can be version-controlled with Git
- Configuration can be batch-generated and modified by script (e.g., auto-add 100 Modbus Tags)

This contrasts sharply with traditional SCADA binary engineering file formats — FUXA's project files are readable, editable, and script-processable pure JSON.

## 6.8 Bridge Role with Commercial SCADA

FUXA's unique role in real projects is as a **Web-native frontend supplement to traditional SCADA**:

- Traditional SCADA's Web publishing is usually extra-cost and limited in features
- FUXA can run independently from traditional SCADA, reading data via OPC UA/Modbus
- In a traditional SCADA + FUXA hybrid architecture: traditional SCADA handles control logic and historical storage; FUXA handles Web visualization and mobile access
- This "frontend separation" architecture enables modernization without replacing existing systems
