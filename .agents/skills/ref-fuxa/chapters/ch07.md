## 7.2 Protocol Adapter Architecture

FUXA's plugin-based protocol architecture aligns closely with AUDESYS HAL's device driver interface:

| FUXA Protocol Adapter | AUDESYS HAL Corresponding |
|----------------------|--------------------------|
| Unified interface: connect / disconnect / read / write | HAL Driver trait |
| Each protocol = independent module (modbus.js / opcua.js / s7.js) | HAL Driver plugins (independent crates) |
| Periodic polling VS event subscription | HAL polling mode and subscription mode |
| Device configuration (host/port/unitID, etc.) | HAL DriverConfig struct |

**FUXA's inspiration for AUDESYS driver architecture**:
- FUXA uses mature npm protocol libraries (e.g., node-opcua); AUDESYS can use mature Rust crates
- FUXA's protocol adapters are loosely coupled; AUDESYS should maintain independence between Drivers
- FUXA's WebAPI driver (HTTP data source) demonstrates: **data sources need not be physical devices — they can be REST APIs**

## 7.3 Lightweight Deployment — AUDESYS Runtime Reference

FUXA's Raspberry Pi deployment experience provides reference for AUDESYS Runtime lightweight targets:

- **Node.js memory usage**: ~200-500 MB on RPi (including Node.js runtime)
- **SQLite as config storage**: No independent database installation needed
- **Docker**: Easy edge device deployment and updates
- **Daemon mode**: Headless device background execution

AUDESYS Runtime's Rust choice provides inherent advantages over Node.js:
- Lower memory (10-50 MB vs 200-500 MB)
- Faster startup (ms vs s)
- No GC pauses (deterministic real-time performance)

FUXA proves that **Web-native HMI is practically viable**:

| Challenge | FUXA's Solution | AUDESYS Reference |

| SVG editor complexity | Self-developed lightweight SVG editor | Reference architecture, or consider open-source editors (svg.js, Fabric.js, Konva.js) |

## 7.5 Tech Stack Selection Lessons

FUXA's tech stack (TypeScript/JavaScript + Node.js + Angular + SQLite) demonstrates the pros and cons of "all-JS stack" in industrial SCADA:

**Advantages**:
- Unified language lowers team skill requirements
- Rich npm ecosystem, fast access to feature libraries
- Frontend-backend code shares type definitions

**Disadvantages**:
- Node.js GC pauses can cause real-time data acquisition jitter
- Large-scale data processing performance insufficient (vs Rust/C++)
- Higher memory usage
- Single-thread model scalability limits

**AUDESYS reference**:
- **Frontend**: TypeScript + React (leverage npm ecosystem)
- **Backend/real-time**: Rust (deterministic real-time, no GC, memory safety)
- **FFI bridge**: Rust core communicates with TypeScript frontend via WebSocket/HTTP/FlatBuffers

## 7.7 Upgrade Path from FUXA Limitations to AUDESYS

Convert FUXA's limitations into AUDESYS's differentiating advantages:

| FUXA Limitation | AUDESYS Solution |
|-----------------|-----------------|
| Node.js single-thread | Rust multi-threading + async task scheduling |
| No real-time determinism | Rust + SCHED_FIFO real-time scheduling |

## 7.8 Pitfalls to Watch

1. **SVG editor complexity trap**: FUXA's SVG editor looks simple but complete implementation (snap, rotate, scale, path editing, export) requires substantial work. AUDESYS should consider building on mature Web graphics libraries (Fabric.js, Konva.js).
2. **Node.js version dependency**: FUXA migrated from Node.js 14 → 18; some S7 libraries (node-snap7) require additional build dependencies. AUDESYS's Rust choice avoids runtime version issues.
3. **SQLite performance ceiling**: FUXA defaults to SQLite for historical data; performance bottlenecks appear at >1000 tags. AUDESYS should consider pluggable time-series storage backends.
4. **Single-process scalability limits**: FUXA is single-process, cannot scale horizontally. AUDESYS Runtime should support multi-process/distributed architecture from the start.
